### Задача 2: Остатки на собственных складах продавца ❌

**Приоритет**: 🔴 Критический (требование Блок 2.1)
**Ответственный**: финдиректор + 1C + разработчик

**Проблема**: нужны остатки на складах компании (не WB):
- СР — (название склада/канала)
- ГГ — (название склада/канала)
- ФР — (название склада/канала)
- Склад ИМ + Внуково — интернет-магазин + Внуково

**Текущие данные**: `stocks_daily_warehouses` в `wb-sales.db` содержит только склады WB (Коледино, Электросталь и т.д.)

**Варианты решения**:

| Вариант | Источник | Сложность |
|---------|----------|-----------|
| A. API 1C | Регистр остатков 1C (если есть REST API) | Средняя |
| B. Выгрузка из 1C | CSV/Excel отчёт → загрузка в SQLite | Низкая |
| C. API ERP | Если используется ERP-система с REST API | Зависит от ERP |

**Что нужно от финдиректора**:
1. Расшифровка аббревиатур: СР = ?, ГГ = ?, ФР = ?
2. Есть ли API для получения остатков по складам? Если да — URL, формат, авторизация
3. Если нет API — можно ли настроить автоматическую выгрузку из 1C (CSV/Excel)?
4. Гранулярность: остатки по артикулам или по артикулам + размеры?

**Схема данных** (предварительная):

```sql
CREATE TABLE IF NOT EXISTS seller_stocks (
    snapshot_date   TEXT    NOT NULL,
    vendor_code     TEXT    NOT NULL,
    warehouse_code  TEXT    NOT NULL,     -- СР, ГГ, ФР, ИМ, Внуково
    warehouse_name  TEXT    DEFAULT '',
    quantity        INTEGER DEFAULT 0,    -- остаток штук
    updated_at      TEXT    DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (snapshot_date, vendor_code, warehouse_code)
);
```

### Задача 3: Параметры BI-отчёта PowerBI

**Приоритет**: 🟡 Средний (после реализации утилиты)
**Ответственный**: финдиректор + BI-аналитик

Утилита предоставляет сырые данные. Следующие расчёты должны быть реализованы в PowerBI
через меры (DAX) или вычисляемые столбцы:

#### 3.1. Параметр СПП по умолчанию

**Тип**: Выбор из списка или ручной ввод
**Значения**: 14%, 20%, 25%, 30%, 35%, 40% (на усмотрение финдиректора)
**Применение**:
- `ОЭК с СПП = onec_base_price * (1 - СПП_по_умолчанию / 100)`
- Используется для расчёта #13 (отклонение СР от ОЭК с СПП)

**Рекомендация**: реализовать как параметр What-If в PowerBI.

#### 3.2. Параметр "Кошелёк WB"

**Тип**: Выбор из списка
**Значения**: 2%, 3%, 5%
**Применение**:
- `РЦ WB = discounted_price * (1 - avg_wb_spp_3d / 100) * (1 - wallet / 100)`
- Используется для расчёта #7 и #12

#### 3.3. Параметр "Скидка Яркие вместе"

**Тип**: Выбор из списка
**Значения**: 5.5%, 10%, 13.5%, 19%, 25%, 50%
**Логика**:
- Если `is_special_price = 1` → скидка НЕ применяется, цена = `onec_sr_price`
- Если `is_special_price = 0` → цена = `onec_sr_price * (1 - discount / 100)`
- Используется для расчёта #11, #12, #13

#### 3.4. Коридоры отклонений

**Тип**: Настраиваемые пороги фильтрации
**Значения (по умолчанию)**:

| Процентный коридор | Абсолютный коридор |
|-------------------|-------------------|
| до 3% | до 100₽ |
| от 3% до 7% | от 100₽ до 500₽ |
| от 7% | от 500₽ |

**Применение**: фильтрация строк + подсчёт артикулов в каждом коридоре.
Рекомендация: реализовать через параметризованные меры DAX с возможностью настройки порогов.

## Справочник: формулы BI-расчётов

Для BI-аналитика — формулы на основе данных из `price_comparison`:

| Показатель | Формула | Исходные колонки |
|-----------|---------|-----------------|
| Отклонение WB от ОЭК % | `(discounted_price - onec_base_price) / onec_base_price * 100` | Уже в `diff_discounted_pct` |
| ОЭК с СПП | `onec_base_price * (1 - [СПП_по_умолчанию] / 100)` | `onec_base_price` + What-If |
| РЦ WB с СПП + кошелёк | `discounted_price * (1 - avg_wb_spp_3d / 100) * (1 - [Кошелёк_WB] / 100)` | `discounted_price`, `avg_wb_spp_3d` + What-If |
| Цена СР с лояльностью | `IF(is_special_price = 1, onec_sr_price, onec_sr_price * (1 - [Скидка_ЯВ] / 100))` | `onec_sr_price`, `is_special_price` + What-If |
| Отклонение СР от РЦ WB | `[Цена_СР_с_лояльностью] - [РЦ_WB_с_СПП_и_кошельком]` | Разность мер |
| Отклонение СР от ОЭК | `[Цена_СР_с_лояльностью] - [ОЭК_с_СПП]` | Разность мер |

## Связь с сервисом indeepa

Утилита не интегрируется с indeepa напрямую — indeepa управляет ценообразованием на WB,
а наша утилита контролирует **результаты** этого ценообразования (сравнивает фактические цены).

Данные для контроля:
1. `product_prices` — цены, которые indeepa установил (фактические)
2. `onec_prices` — цены из 1C (эталон для сравнения)
3. `sales.ppvz_spp_prc` — фактический СПП WB (побочный эффект ценообразования)
