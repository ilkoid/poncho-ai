# Анализ утилиты compare-wb-1c-prices: проблемы расчёта СПП

**Дата:** 2026-04-10
**Утилита:** `cmd/data-analyzers/compare-wb-1c-prices/`
**Источник данных:** `db/wb-sales.db` → `bi.db` (таблица `price_comparison`)
**Заказчик:** Финансовый директор (PowerBI отчёт)

---

## 1. Текущая логика расчётов

### 1.1. Цены из источников

| Поле | Источник | Описание |
|------|----------|----------|
| `onec_base_price` | 1C `onec_prices` (тип "Розничная цена ОЭК") | Розничная цена без скидок |
| `onec_spp25_price` | Вычисляется: `onec_base_price * 0.75` | ОЭК с захардкоженным СПП 25% |
| `onec_sr_price` | 1C `onec_prices` (тип "Розничная цена СР") | Цена для розничных магазинов |
| `onec_special_price` | 1C `onec_prices` (тип "Спец цена для акции") | Флаг акции (цена=1 → активна) |
| `wb_price` | WB API `product_prices.price` | Розничная цена WB (без скидки) |
| `wb_discounted_price` | WB API `product_prices.discounted_price` | Цена со скидкой продавца |
| `wb_discount_pct` | WB API `product_prices.discount` | Скидка продавца % |
| `wb_club_price` | WB API `product_prices.club_discounted_price` | Цена WB Club |
| `avg_wb_spp_3d` | Вычисляется: `AVG(ppvz_spp_prc)` из `sales` за 3 дня | Реальный СПП WB из продаж |
| `avg_wb_spp_assortment` | Вычисляется: `AVG(ppvz_spp_prc)` по всем товарам | Средний СПП по ассортименту |

### 1.2. Формулы отклонений (текущие)

```
diff_base       = wb_price - onec_base_price
diff_base_pct   = diff_base / onec_base_price * 100

diff_discounted     = wb_discounted_price - onec_spp25_price
diff_discounted_pct = diff_discounted / onec_spp25_price * 100
```

Где `onec_spp25_price = onec_base_price * 0.75` (захардкождено, query.go:127).

### 1.3. Источник СПП (текущая логика)

```
avg_wb_spp_3d = COALESCE(spp.avg_spp_3d, oa.avg_spp_assortment)
```

- Если есть продажи товара за 3 дня → используется средний СПП по этому товару
- Если продаж нет → fallback на средний СПП по всему ассортименту
- **Пользовательский SPP (spp25_percent из конфига)** используется ТОЛЬКО для `onec_spp25_price`, но НЕ как fallback для отсутствующих продаж

## 2. Маппинг проблем на код

### Проблема 1: "СПП задаваемый влияет на Отклонение СР от РЦ WB, а не на Отклонение СР от ОЭК с СПП"

**Файл:** `main.go:221`
```go
diffDiscounted := s.WBDiscountPrice - s.OneCSPP25Price
```

**Причина:** `OneCSPP25Price` вычисляется как `onec_base_price * (1 - 25/100)` — это **ОЭК с СПП**, а не **РЦ WB**. Поэтому изменение `spp25_percent` в конфиге меняет `diff_discounted`, что PowerBI показывает как "Отклонение СР от ОЭК с СПП".

Однако, если PowerBI показывает этот параметр в другом месте отчёта (как "Отклонение СР от РЦ WB"), то проблема в маппинге колонок PowerBI, а не в расчётах Go.

**Вывод:** Требуется уточнение — какие именно колонки в PowerBI соответствуют каким полям в `price_comparison`.

### Проблема 2: "Цена ОЭК с СПП задаваемой не выведена"

**Файл:** `query.go:127`
```sql
ROUND(op.price * 0.75, 2) AS onec_spp25_price
```

**Причина:** Поле `onec_spp25_price` использует **захардкожденный коэффициент 0.75** (т.е. СПП=25%). Нет отдельного поля, которое считает:
```
onec_price_with_spp = onec_base_price * (1 - effective_spp / 100)
```
где `effective_spp` — это реальный СПП из WB продаж (с фолбэком на пользовательский).

**Статус:** Поле `onec_spp25_price` есть, но оно не отражает реальный СПП. Нужна новая колонка.

### Проблема 3: "Отклонение СР от ОЭК с СПП % не меняется при изменении никаких параметров"

**Файл:** `main.go:227-229`
```go
if s.OneCSPP25Price > 0 {
    diffDiscPct = (diffDiscounted / s.OneCSPP25Price) * 100
}
```

**Причина (подтверждено):** `OneCSPP25Price` = `onec_base_price * 0.75` — это константа, которая **не зависит** от:
- `avg_wb_spp_3d` (реальный СПП из продаж)
- `spp25_percent` (пользовательский SPP в конфиге) — влияет только через пересчёт утилиты, не динамически
- `onec_special_price` (скидка по программе лояльности)

Поле `diff_discounted_pct` всегда считает через один и тот же знаменатель, поэтому визуально "не меняется".

**Статус:** 100% баг. Нужно пересчитывать через `effective_spp`.

### Проблема 4: "Розничная цена WB с СПП должно считаться через СПП WB из API"

**Файл:** `storage.go:59-67` — схема БД

**Причина:** В таблице `price_comparison` нет поля:
```
wb_price_with_spp = wb_price * (1 - effective_spp / 100)
```

Есть только `wb_discounted_price` из WB API (скидка продавца), но нет расчётной цены с учётом СПП.

**Статус:** Нужна новая колонка `wb_price_with_spp`.

## 3. Требуемая бизнес-логика (по описанию заказчика)

### 3.1. Эффективный СПП

```
effective_spp = {
    avg_wb_spp_3d,                     если есть продажи за 3 дня (spp_source = 'product')
    spp_user (пользовательский SPP),    если продаж нет
}

spp_type = {
    'wb_api'         — если есть продажи, используется СПП из WB
    'user_fallback'  — если продаж нет, используется пользовательский СПП
}
```

**Важно:** Ассортиментный средний СПП (`avg_wb_spp_assortment`) больше НЕ используется как fallback — вместо него берётся пользовательский SPP.

### 3.2. Новые расчётные поля

```
ОЭК_с_СПП     = onec_base_price * (1 - effective_spp / 100)
WB_цена_с_СПП  = wb_price * (1 - effective_spp / 100)
```

### 3.3. Отклонения (исправленные)

```
Отклонение СР от ОЭК с СПП     = wb_discounted_price - ОЭК_с_СПП
Отклонение СР от ОЭК с СПП %   = Отклонение / ОЭК_с_СПП * 100
```

### 3.4. Сравнение с другими маркетплейсами (будущее)

Пользовательский SPP нужен для расчёта цен на Ozon/Яндекс:
```
Цена_на_другом_маркете = onec_base_price * (1 - spp_другого_маркета / 100)
```

На первом этапе — только WB, с одним пользовательским SPP.

## 5. Вопросы заказчику

### В1. Приоритет источников СПП

Текущая логика использует трёхуровневый fallback: товарный СПП → ассортиментный → нет данных.

По описанию: "СПП WB из API, если данных нет, то через заданный СПП пользователем".

**Вопрос:** Подтвердите порядок приоритета:
1. СПП из продаж WB за 3 дня (товарный) — приоритет 1
2. Пользовательский СПП — приоритет 2 (если продаж нет)

Ассортиментный средний СПП (`avg_wb_spp_assortment`) исключается из расчётов?

### В2. WB Discounted Price vs WB цена с СПП

Поле `wb_discounted_price` из WB API — это цена **со скидкой продавца** (та, что видит клиент в карточке товара без учёта персональных скидок).

СПП (скидка постоянного покупателя) — это **дополнительная** скидка поверх скидки продавца, которая применяется в корзине/при оформлении заказа.

**Вопрос:** Верно ли, что:
- `wb_discounted_price` = цена для клиента **без СПП** (скидка продавца)
- `wb_discounted_price * (1 - spp/100)` = цена для клиента **с СПП** (итоговая)

Именно итоговая цена (с СПП) должна сравниваться с ОЭК?

### В3. "Отклонение СР от РЦ WB" — что это?

**Вопрос:** "Отклонение СР от РЦ WB" в отчёте PowerBI — это:
- **Вариант А:** `wb_discounted_price - wb_price` (размер скидки WB в рублях, всегда отрицательное)
- **Вариант Б:** `wb_price - onec_base_price` (разница розничных цен WB vs 1C, текущий `diff_base`)
- **Вариант В:** что-то другое?

Нужно точное определение для правильного маппинга.

### В4. СПП для других маркетплейсов

**Вопрос:** На первом этапе — один пользовательский SPP для всех маркетплейсов, или нужен отдельный параметр для Ozon/Яндекс? Если отдельный — какие значения по умолчанию?

## Приложение А: Схема данных

### Источник (wb-sales.db)

```
onec_goods          → onec_prices (по guid + snapshot_date + type_name)
                    → cards (по article = vendor_code)
                    → product_prices (по nm_id + snapshot_date)
                    → products (по nm_id)
                    → sales (по nm_id, для СПП)
                    → pim_goods (по article = identifier)
```

### Результат (bi.db)

```
price_comparison:
  48 колонок (идентификация, иерархия, атрибуты, цены 1С, цены WB, отклонения, статусы, СПП)
  UNIQUE(vendor_code, nm_id, wb_snapshot_date, onec_snapshot_date)
```

### Ключевые поля WB для расчёта СПП

Таблица `sales`, поле `ppvz_spp_prc` — фактический процент СПП из каждой продажи. Усредняется за 3 дня.
