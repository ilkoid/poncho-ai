# fix-scrub-substring

YAML-управляемая утилита, которая **во всех текстовых колонках всех таблиц** PostgreSQL-схемы
заменяет каждое вхождение `read_substring` на `write_substring` (поиск **без учёта регистра**,
замена **дословно** как в YAML). Цель — скрыть чувствительные данные (например, бренд) в
кеш-базе PG с данными WB перед тем, как отдавать её нейросетям.

Утилита построена по принципу **«никаких дефолтов»**: если обязательное поле YAML отсутствует —
она падает с понятной ошибкой, а не подключается молча к неправильной базе. Поэтому `config.yaml`
содержит ВСЕ параметры подключения явно.

---

## Режимы

| Флаг | Режим БД | Что делает |
|---|---|---|
| `--mock` | **нет коннекта** | печатает синтетический план (проверка проводки без БД) |
| `--show` | read-only | какие таблицы/колонки совпадают + счётчики матчей |
| `--dry-run` | read-only | плановые UPDATE + пары «до→после» (samples), **без записи** |
| `--apply` | **деструктивный** | выполняет все UPDATE в **одной транзакции** (all-or-nothing) |
| `--select_tables=a,b` | любой | ограничить набор таблиц (комбинируется с любым режимом) |

`--show`/`--dry-run` открывают `BEGIN READ ONLY` транзакцию — баг в запросе не может мутировать данные.
`--apply` — единственный гейт записи.

`--config` по умолчанию `config.yaml` (локальный конфиг в папке утилиты).

```bash
# 1. Из корня репо перейти в папку утилиты
cd cmd/fix-utilities/fix-scrub-substring

# 2. Экспортировать пароль PG (config.yaml: password: "${PG_PWD}")
```

## Сценарии (копи-паста)

### 0. Каждый раз перед запуском
```bash
cd cmd/fix-utilities/fix-scrub-substring
export PG_PWD='пароль'
```

### 1. Офлайн-проверка проводки (нет коннекта к БД)
```bash
go run . --mock
```
Печатает синтетический план с префиксом `[MOCK]`. Подключение к БД **не открывается**.
Удобно проверить, как отработают фильтры таблиц и формат вывода, не трогая данные.

### 2. Где живёт подстрока (read-only)
```bash
go run . --show
```
Выводит таблицу `TABLE | COLUMN | TYPE | MATCHES` — в каких колонках сколько совпадений.
Это **один консолидированный scan на таблицу** (counts по всем колонкам через `FILTER`), а не
по одному scan'у на колонку — критично для таблиц на десятки млн строк.

### 3. Только конкретные таблицы (read-only)
```bash
go run . --show --select_tables=cards,products,onec_goods
```
`--select_tables` пересекается с `include_tables` из конфига; `exclude_tables` вычитается.
Работает в любом режиме. Удобно сузить область перед `--apply`.

### 4. План замен с примерами «до→после» (read-only, **без записи**)
```bash
go run . --dry-run
```
Показывает плановые UPDATE, число затронутых строк и по `sample_rows` пар «до→после» на колонку.
**Обязательно прогонять перед `--apply`** — убедиться, что замена выглядит правильно
(особенно множественные вхождения и разные регистры в одном поле).

### 5. Dry-run по подмножеству таблиц
```bash
go run . --dry-run --select_tables=cards
```

### 6. Выполнить скраб (**деструктивно**, единая транзакция)
```bash
go run . --apply
```
Все UPDATE по всем таблицам — в **одной atomic-транзакции**: либо бренд скрыт везде, либо нигде
(при ошибке на любой таблице откатывается всё). Прогресс печатается по таблицам с elapsed.
Запуск против `wb_data_prod` — **только вручную**, после сверки `--dry-run`.

### 7. Применить только к выбранным таблицам
```bash
go run . --apply --select_tables=cards,feedbacks
```
Удобно для поэтапного скраба или повторного прохода по конкретным таблицам.

### 8. Включить scrub jsonb/json колонок
В `config.yaml` поставить `include_jsonb: true`, затем любой режим:
```bash
go run . --show      # теперь видны и jsonb-колонки
go run . --apply     # jsonb-значения скрабятся через ::text → regexp_replace → ::jsonb
```
> В текущем кеше `wb_data_prod` jsonb/JSON колонок **нет** (замер 2026-06-17), поэтому для этой
> базы путь — no-op. Оставлен для будущих баз. При скрабе jsonb невалидный результат (если замена
> сломает JSON-структуру) вызовет ошибку и откат транзакции.

## Конфиг `config.yaml`

| Поле | Обяз. | Описание |
|---|---|---|
| `pg.host` | ✅ | хост PostgreSQL |
| `pg.port` | ✅ | порт (1..65535) |
| `pg.user` | ✅ | пользователь |
| `pg.password` | ✅ | пароль; пишите `"${PG_PWD}"` — раскрывается из env |
| `pg.database` | ✅ | имя базы |
| `pg.sslmode` | ✅ | `disable`/`allow`/`prefer`/`require`/`verify-ca`/`verify-full` |
| `schema` | ✅ | схема (обычно `public`) |
| `read_substring` | ✅ | что искать (без учёта регистра) |
| `write_substring` | ✅ | на что менять (дословно; **пустое запрещено**) |
| `column_types` | ✅ | список `information_schema.data_type` (`text`, `character varying`, `character`) |
| `include_jsonb` | — | `true` добавляет `jsonb`/`json` к `column_types` (default `false`) |
| `include_tables` | — | white-list таблиц (сужает область) |
| `exclude_tables` | — | black-list таблиц (всегда пропускаются) |
| `sample_rows` | — | пар «до→после» на колонку в `--dry-run` (default `3`) |
| `statement_timeout_seconds` | — | таймаут транзакции, сек (default `0` = без лимита) |

Удалите/оставьте пустым любое обязательное поле — утилита упадёт с понятной ошибкой
(`pg.host is required`, `write_substring must not be empty`, …), а не молча подключится не туда.

## Как это работает (кратко)

- **Поиск без учёта регистра, замена дословно.** `read_substring` матчит все регистровые варианты
  (`PlayToday`/`PLAYTODAY`/`playtoday`); замена — ровно `write_substring` как в YAML.
  Кириллица и латиница case-fold корректно (проверено на живой БД).
- **Plain vs regex.** Если в `read_substring` нет regex-метасимволов (обычный случай для бренда),
  фильтр скана использует быстрый `ILIKE` вместо POSIX `~*`. Замена всегда через `regexp_replace('gi')`
  (другого способа case-insensitive global-replace в PG нет). Метасимволы в названии → авто-фолбэк на `~*`.
- **Консолидация по таблицам.** Count и UPDATE строятся **на всю таблицу сразу** (все её колонки в одном
  запросе через `count(*) FILTER (…)` / `SET col=CASE WHEN … THEN regexp_replace(…) ELSE col END … WHERE … OR …`).
  Heap читается 1 раз на таблицу, MVCC-версия создаётся 1 на совпадающую строку — вместо K сканов и до K
  версий на строку при поколоночной схеме. На десятках млн строк это главный выигрыш.
- **`statement_timeout`.** Пул навязывает 5-минутный таймаут; каждый tx утилиты снимает его через
  `SET LOCAL statement_timeout` (значение из `statement_timeout_seconds`, default 0 = без лимита) — иначе
  большой консолидированный UPDATE на десятках млн строк убился бы по таймауту и откатил весь скраб.
- **Атомарность.** Все UPDATE — в одной транзакции. Ошибка на любой таблице → откат всего скраба
  (проверено e2e). Частичной утечки не бывает.

## Безопасность

- **`--apply` против `wb_data_prod` — только вручную.** Всегда сначала `--dry-run` и сверить выборки.
- **Read-only режимы** (`--show`/`--dry-run`) используют `BEGIN READ ONLY` — мутировать данные физически не могут.
- Утилита подключается **только к PostgreSQL** через `config.yaml`. Она не имеет доступа к `/var/db/*.db`
  (SQLite-prod) и к WB API — архитектурно.
- Идентификаторы (schema/table/column) квотятся; подстроки идут только как параметры (`$1`/`$2`/`$3`),
  никогда интерполяцией — SQL-инъекции со стороны данных нет.
- `--mock` **не** открывает коннект к БД — безопасен для проверки проводки.

## Типовые проблемы

| Симптом | Причина / решение |
|---|---|
| `config validation failed: pg.password is required` | не экспортирован `PG_PWD` (или пустой) |
| `connect postgres: …` / `No route to host` | хост/порт из `config.yaml` не маршрутизируются — проверьте `pg.host`/`pg.port` |
| `specify one of --show, --dry-run, --apply (or --mock)` | не передан ни один action-флаг |
| `write_substring must not be empty` | пустая замена запрещена (защита от удаления данных) |
| `--apply` упал с `SQLSTATE …` | транзакция откатилась целиком — данные не изменены; исправьте причину и перезапустите |
